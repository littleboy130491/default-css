On the Quiet Craft of Good Defaults

An argument for spending an afternoon on the settings file before writing a single component.

Design System Prose

Most design systems fail not at the edges but at the origin — in the handful of values that get quietly inherited by everything else. The work of choosing those values well is rarely visible, and that is precisely why it pays off.

There is a particular kind of website that feels considered the instant it loads. The line-height is correct. The headings sit at a measured width. The accent color is not blue. Nothing shouts. You scroll, and the page rewards the scroll. You do not need to be a designer to feel it, and you do not need to be a designer to build it — you only need to take the defaults seriously.

The settings file is the product

A stylesheet is a kind of argument. Every variable is a sentence about what the project thinks matters. When the default --color-bg is #ffffff, the project is saying something (probably by accident) about clinical neutrality. When it is #faf8f4, it is saying something else — warmth, paper, a little restraint.

Good tokens do three things at once: they carry a decision, they carry a reason, and they carry an invitation to override. The best settings files read less like configuration and more like the first chapter of a style guide that happens to compile.

Designers love to say that constraint breeds creativity. They are less eager to admit that most constraint is set by whoever named the first variable.

A working checklist

Before writing a single component, it is worth spending an hour on the following — in this order, because each depends on the one before it:

  1. Pick a background that is not pure white. The rest of the palette will thank you.
  2. Pick an accent that is not the default framework blue.
  3. Set a measure. Sixty-five characters is a good measure. Forty-eight is a better measure for essays.
  4. Decide how much air goes between a heading and the paragraph that follows it. Then halve it.
  5. Give your buttons a weight heavier than you think they need.

None of these are opinions about your project. They are opinions about the kind of project that looks good before anyone touches it.

On restraint

A list of things the default stylesheet should not include:

  • A gradient.
  • A drop shadow larger than 0 8px 24px.
  • More than one accent color.
  • A border radius above 0.5rem, except on pills.
  • Any font loaded only because it was trending this quarter.

Add these when the design calls for them. Not before.

Typography is a system, not a choice

Choosing a typeface is the easy part. The hard part is deciding what that typeface means at every size. A display serif at 88px is an entirely different object than the same serif at 22px — the optical size axis exists because the designers of the font already knew this. Most projects do not.

A simple heuristic: if your h1 and your h3 use the same tracking value, one of them is wrong. Tighter as you go larger; looser as you go smaller. It is not a rule so much as a ramp. 

/* A tracking ramp, roughly */
--tracking-h1: -0.04em;
--tracking-h2: -0.02em;
--tracking-h3: -0.01em;
--tracking-p:   0;
--tracking-eyebrow: 0.08em;

The values are less important than the direction. The ramp is the idea.

On finishing

A design system is never done, but a settings file can be. There is a version of it where every value is defensible, every token is used, and the next person to open the project can ship a page without asking a single question. That version is worth reaching.
